# Evaluation Grid — Rubrica di Valutazione Homepage

## Come usare questa rubrica

Per ogni sezione presente nella homepage, assegna uno score:
- ✅ **Funziona** — fa esattamente quello che deve fare
- 🆗 **Migliorabile** — presente ma incompleto o debole
- ❌ **Non funziona** — assente, sbagliato, o controproducente

Inizia sempre verificando la coerenza strategica (anchor) prima di valutare le singole sezioni.

---

## 0. Verifica Strategica (Prima di tutto)

### 0.1 Anchor Identification
- [ ] Quale anchor sta usando la homepage? (Competitive Alternative / Product Category / Use Case / Activity)
- [ ] L'anchor è dichiarato esplicitamente nella hero o è ambiguo?
- [ ] L'anchor scelto è giustificato dal contesto (stage startup, maturità mercato, awareness buyer)?

### 0.2 Coerenza dell'Anchor
- [ ] Tutte le sezioni usano lo stesso anchor? O c'è un mismatch tra hero (use case) e capabilities (category)?
- [ ] Il livello di awareness assunto dal copy corrisponde all'awareness reale del buyer?
- [ ] Se l'anchor è "Competitive Alternative": il competitor nominato è davvero dominante nel mercato?

### 0.3 ICP Chiarezza
- [ ] Si capisce immediatamente per chi è il prodotto?
- [ ] Il copy parla a una persona specifica o a "tutti"?
- [ ] Un visitatore fuori target capirebbe che il prodotto non è per lui?

## 2. Hero Section

### Elementi presenti (checklist 7 elementi)

| Elemento | ✅ | 🆗 | ❌ | Note |
|----------|----|----|-----|------|
| Product Category (cos'è) | | | | |
| Capabilities / Use Case (cosa posso fare) | | | | |
| Benefits (qual è il valore) | | | | |
| ICP (è per me?) | | | | |
| Product visual (immagine/video UI) | | | | |
| Social proof / trust | | | | |
| Objection handling | | | | |

### Qualità del copy

| Criterio | ✅ | 🆗 | ❌ |
|----------|----|----|-----|
| H1: max 8-10 parole, chiaro senza contesto | | | |
| H1 derivato da problem/outcome (non da feature) | | | |
| Subheadline: combina feature + benefit | | | |
| ICP dichiarato nell'H1 o subheadline | | | |
| CTA primario chiaro e specifico | | | |
| Un visitatore capisce in 5 secondi cos'è il prodotto | | | |

### Domande diagnostiche
- L'H1 parla del problema/outcome dell'utente o descrive il prodotto?
- Il copy è scritto in prima persona ("We help you...") o dal punto di vista del buyer ("You can now...")?
- La social proof nella hero è sopra o sotto il fold? (dovrebbe essere above the fold)

## 3. Social Proof 1 (Logo Bar)

| Criterio | ✅ | 🆗 | ❌ |
|----------|----|----|-----|
| Loghi riconoscibili per il target ICP | | | |
| Tagline che definisce l'ICP ("Trusted by 500+ B2B SaaS founders") | | | |
| Posizionata subito dopo la hero | | | |
| Numero di clienti o metrica di scala quando disponibile | | | |

**Errore comune:** Loghi di aziende irrilevanti per il target (es. una startup che mostra loghi enterprise quando vende a PMI, o viceversa).

## 4. Problem Section

| Criterio | ✅ | 🆗 | ❌ |
|----------|----|----|-----|
| Descrive lo status quo (come il buyer fa le cose oggi) | | | |
| Identifica le limitazioni del current way | | | |
| Articola le conseguenze negative (il costo dell'inazione) | | | |
| Il visitatore ICP si riconosce | | | |
| Il visitatore non-ICP capisce che non è per lui | | | |
| Copy empatico, non accusatorio | | | |

**Quando includere:** Solo se il buyer non è già problem-aware. Se il mercato è maturo, salta questa sezione o usa una variante "old way vs new way".

**Varianti layout:**
- 3 colonne di agitazione (una per pain point)
- Old Way vs New Way (2 card affiancate con X e ✓)
- Equazione visiva: [Problema A] + [Problema B] = [Disastro]

## 6. Capabilities / Benefits Section

| Criterio | ✅ | 🆗 | ❌ |
|----------|----|----|-----|
| Max 3 capability principali (non un elenco di tutte le feature) | | | |
| Ogni capability ha: titolo orientato al beneficio + feature specifica + visual del prodotto | | | |
| Il titolo H3 è scritto come beneficio/azione per l'utente (non come descrizione del prodotto) | | | |
| Il visual mostra la feature in azione (non un'immagine decorativa) | | | |
| La logica del Reversal è rispettata (benefit = opposto del problema dichiarato) | | | |

**Domande diagnostiche:**
- Se togli le immagini, il testo da solo è sufficiente a capire il valore?
- Il titolo H3 inizia con un verbo d'azione o con il nome del prodotto?
- Le 3 capability sono distinte tra loro o si sovrappongono?

---

## 7. Special Purpose Section (se presente)

Valuta in base al tipo:

**Feature Wall:**
- Copre le funzionalità importanti senza sovraccaricare?
- Usa gerarchia visiva (funzionalità primarie vs secondarie)?

**How It Works:**
- Max 3-5 step?
- Ogni step ha un risultato concreto (non solo un'azione)?

**Use Cases:**
- Coprono scenari che il buyer riconosce?
- Ogni use case ha un link a una pagina dedicata?

**Security/Compliance:**
- I badge sono reali e aggiornati?
- C'è una frase che spiega il beneficio (non solo i loghi delle certificazioni)?

## Clarity Test Finale

Prima di chiudere l'analisi, applica questi test rapidi:

- [ ] **Test 5 secondi**: Un visitatore che non conosce il prodotto capisce cos'è e per chi è in 5 secondi?
- [ ] **Test nonna/ubriaco**: Una persona non tecnica navigherebbe il sito senza confondersi?
- [ ] **Test ICP**: Un buyer ideale direbbe "questo è esattamente per me"?
- [ ] **Test non-ICP**: Un buyer non ideale direbbe "questo non fa per me"?
- [ ] **Test coerenza**: Tutte le sezioni raccontano la stessa storia con lo stesso anchor?
- [ ] **Test copy elementare**: Il testo è comprensibile a livello di lettura di quinta elementare?
- [ ] **Test CTA unico**: C'è un solo CTA primario dominante o la pagina fa troppe richieste contemporaneamente?
